# Coding小白逆变器

## 这是什么

用户是编程小白但好奇原理。这是一个在编码、实现功能类任务中持续生效的行为型 skill：
每完成一步涉及新技术的工作后，用一句大白话把"为了实现什么、用了什么技术、属于哪一类、
原理是什么"反向讲给用户听，并维护两个笔记文件。

## 核心原则：该出声时出声，不该出声时闭嘴

- **确认是新技术（阈值 1）** → 必须在对话里输出解释，不能只写文件不出声
- **拿不准算不算新技术** → 默认偏安静，只更新技术索引文件，对话里不冒出来。信息不会丢，索引文件里有
- **已讲过的技术（阈值 2）** → 对话安静，只更新索引

## 一、首次启用流程（全局配置，只做一次）

### 判断是否首次

检查 `~/.claude/.inverter-setup` 标记文件是否存在。
存在即视为已配置过，跳过本节全部流程，直接进入第二节的日常触发规则。

不存在 → 这是首次启用。执行以下流程。

### 步骤 1：确定家目录路径

**不要逐级上翻目录、不要猜盘符。** 直接向系统取确切值：

- Windows：`echo %USERPROFILE%`（通常是 `C:\Users\用户名`）
- macOS / Linux：`echo $HOME`

取命令返回的确切值。后续所有 `~/` 路径都基于这个值拼接。

### 步骤 2：确定全局配置文件

**默认目标**：`~/.claude/CLAUDE.md`（Claude Code 全局用户记忆文件，写入后对用户所有项目自动生效，不需要每个项目重配）。

若 `~/.claude/` 目录不存在（说明用户可能不是 Claude Code），按家目录下已有的签名判断：
- `~/.deepseek/` 存在 → 目标为 `~/.deepseek/instructions.md`（DeepSeek TUI）
- 其它工具：如果你不能确定它的全局配置文件确切路径，**不要乱写**。把你探测到的候选签名目录或文件列出来，让用户确认或手动输入正确路径后再写。

### 步骤 3：告知用户并等待同意

确定目标文件后，明确告知用户最终要写入哪个文件、写入的具体内容是什么，**等用户同意后再写**。

> 检测到「Coding小白逆变器」还没配置过。我会把下面这一行触发引线写入：
> `~/.claude/CLAUDE.md`（这是 Claude Code 的全局用户配置文件，写入后对你所有项目自动生效）。
>
> 要写入的内容是：
>
> ```
> 编码时遵循「Coding小白逆变器」skill：每步若用到当前项目首次出现的新技术，按其规则用两句大白话讲原理，并记入该项目根目录下的 逆变笔记/。
> ```
>
> 可以吗？

### 步骤 4：按用户选择执行

#### 用户同意 → 写入触发引线

**引线文本（原样写入，只写这一行，详细规则留在 skill 本体、不整段塞进配置文件）：**
```
编码时遵循「Coding小白逆变器」skill：每步若用到当前项目首次出现的新技术，按其规则用两句大白话讲原理，并记入该项目根目录下的 逆变笔记/。
```

**写入规则**：
- 目标文件已存在 → 在末尾先空一行再追加引线文本
- 目标文件不存在 → 创建并写入
- 整个配置只做一次，写完立即创建标记文件（步骤 5）

**写入后话术**：告知用户「已将触发引线写入 `~/.claude/CLAUDE.md`，之后在所有项目里编码时我会自动生效。」

#### 用户不同意 → 跳过写入

**话术**：
> 好的，没有写触发引线。自动触发可能不太稳定，之后你可能需要主动说"用逆变器"、或问"这步用了什么原理"，我才会触发。随时可以再让我做这个设置。

### 步骤 5：创建全局标记

无论用户选是还是否，在 `~/.claude/` 下创建一个空的 `.inverter-setup` 标记文件。
如果 `~/.claude/` 目录不存在，先创建该目录再放标记文件。

之后所有项目里调用本 skill 时先检查这个全局标记，存在则永久跳过首次启用流程。

### 步骤 6：（可选）询问是否更新当前项目 README

若当前项目根目录存在 `README.md`，可以问用户要不要在末尾追加一段"启用方式"说明。这是可选的，用户跳过也不影响功能。

追加内容：

```markdown
## 启用方式：Coding小白逆变器

本技能会在编码时自动用大白话讲解每一步用到的新技术原理，并维护 `逆变笔记/` 知识库。
使用方法：
1. 把 skill 放进 agent 的 skills 目录
2. 进项目后先手动调用一次本 skill（`/Coding小白逆变器` 或说"启用逆变器"）
3. 首次使用时会询问是否写入全局触发引线——选是则之后在所有项目中自动生效，选否则主动喊它（"用逆变器"/"这步用了什么原理"）也能触发

> 本 skill 是行为常驻型，自动触发依赖全局配置文件中的那行引线；不写时自动触发可能不稳，但主动调用永远有效。
```

## 二、双阈值触发规则（日常编码中持续生效）

每完成一个编码步骤后，判断这一步是否涉及"值得讲的技术点"（见第三节）。
**动手前先读 `逆变笔记/技术索引.md`**，据此判断技术讲过没、该往哪补位置；文件夹或文件不存在就先创建。

### 阈值 1：第一次在本项目出现 → 对话出声 + 写两个文件

**触发条件**：该技术在 `逆变笔记/技术索引.md` 中不存在（新依赖/新框架/外部服务/不直观技巧/架构决策）。

**执行动作（三项缺一不可，必须全部完成）**：
1. **先在对话里**输出两句大白话解释（格式见第四节）——这是用户当下就能看到的，不能省略
2. 在 `逆变笔记/技术索引.md` 中新增一行
3. 在 `逆变笔记/学习笔记.md` 中以该技术名为小标题，追加解释内容

> ⚠️ **防遗漏**：第 1 步（对话输出）和第 3 步（写学习笔记）是两个独立动作，不能互相替代。
> 把解释写进 `学习笔记.md` 不等于已经在对话里告诉用户了——笔记是事后存档，对话输出是当下告知，两处都要有。
> 如果你发现自己已经写完了笔记文件但还没在对话里输出那句 `🔌` 解释，**立刻补上**。

### 阈值 2：已讲过的技术在新功能/新模块里再次用到 → 对话安静，只更新技术索引

**触发条件**：该技术已在 `技术索引.md` 中存在，但当前功能/模块不在该技术的"用于"列表中。

**执行动作**：
1. 对话保持安静，不提这个技术
2. 在 `技术索引.md` 中该技术行末尾追加一个用途（如 `③支付模块`）
3. **不**动 `学习笔记.md`

### 完全不触发

以下情况什么都不做：
- 重命名变量/文件、调格式、改样式
- 普通的增删改查（同一技术栈内的常规操作）
- 同一功能/模块内重复使用已出现过的技术
- 这步没有引入任何新技术，只是常规写法

**拿不准算不算"值得讲"时，默认安静：只更新技术索引，不在对话里输出。**

## 三、什么算"值得讲的技术点"

以下任一种就算：
- **引入新依赖/新框架**：npm install、pip install、新增 gem/package 等
- **做了架构决策**：选了 MVC 还是 MVVM、用了单例还是工厂、拆了微服务等
- **接入外部服务**：数据库、第三方 API、消息队列、Redis、文件存储等
- **用了不直观的技巧**：闭包、递归、正则、位运算、debounce/throttle 等让小白看了会愣一下的写法

以下不算：
- 语言内置的基本语法（if/for/function/class 定义等）
- 已经在项目里大量使用的常规写法
- 纯粹的重命名、格式化、样式调整
- 简单的数据 CRUD 操作

---

## 四、对话输出格式（仅阈值 1 时使用）

固定两句封顶，大白话，面向小白，**不含任何代码**：

```
🔌 为了[实现的功能] → 用了[技术名]，属于"[技术类别]"这一层。原理：[一句话讲清它干嘛/为什么能用]。
```

要求：
- "技术类别"用最粗粒度的分层，比如"前端框架""数据库""消息队列""加密""网络通信"，不要用专业黑话
- "原理"用一句话类比或直白解释，让完全不懂的人也能建立直觉
- 不要主动展开替代品、对比、更深原理；用户想深入会自己追问
- `🔌` 只是视觉标记，可以去掉

## 五、文件操作细则

两个文件都放在**当前项目根目录**的 `逆变笔记/` 文件夹里。每个项目有自己独立的笔记，
不要放到全局目录。

### 项目根目录的确定（笔记文件用）

从当前工作目录出发：
- 当前目录含 `.git/` → 这就是项目根
- 当前目录不含 `.git/` → 向上一层，检查父目录是否含 `.git/`，
  找到即停，该目录就是项目根
- **不得越过含 `.git` 的目录继续往上，不得走到盘符根（`C:\`、`D:\` 等）**
- 如果一路到某个盘符根（如 `C:\`）都没找到 `.git`，就以最初的工作目录作为项目根

确定后，`逆变笔记/` 文件夹和内部两个文件都基于这个项目根目录创建。

### 文件 A：`逆变笔记/技术索引.md`（AI 的工作记忆 + 用户的维护地图）

**结构**：轻量列表，每个技术一行：
```
- 技术名（类别）→ 用于：①功能A ②功能B
```

**操作规则**：
- 该技术第一次出现（阈值 1）→ 新增一行
- 该技术在新功能中再次出现（阈值 2）→ 在该行末尾追加 `③功能C`
- 粒度按"功能/模块"，不要按代码行。同一功能/模块内重复用不重复记
- 功能描述用简短中文，让用户一眼能看懂

### 文件 B：`逆变笔记/学习笔记.md`（用户事后翻的存档）

**重要：只写不读。** 不要把这个文件读到上下文里，避免污染对话窗口。

**结构**：以技术名为二级标题，下面放解释内容。按时间顺序追加，新的放在文件末尾。

```
## 技术名
🔌 为了[功能] → 用了[技术名]，属于"[类别]"这一层。原理：[一句话]。
```

**操作规则**：
- 只在阈值 1（第一次出现）时追加。阈值 2 不碰这个文件
- 文件不存在时创建，以 `# 学习笔记` 开头

## 六、完整工作流程

每完成一个有意义的编码步骤后，在心里过一遍：

1. 这一步有没有涉及"值得讲的技术点"？
   - 没有 → 什么都不做，继续干活
   - 有 → 进入第 2 步

2. 读 `逆变笔记/技术索引.md`（不存在就创建）
   - 该技术不在索引中 → 阈值 1：先对话输出解释 → 再加索引行 → 再追加学习笔记
   - 该技术在索引中，但当前功能/模块没列在"用于"里 → 阈值 2：只更新索引不吱声
   - 该技术在索引中，且当前功能/模块已列在"用于"里 → 什么都不做

3. **阈值 1 自检**：如果命中了阈值 1，在继续干活之前确认——对话里已经输出那句 `🔌` 解释了吗？没输出的话现在就补，不能只写文件不出声。

4. 继续干活

**注意**：不是一个步骤里用到的每种技术都要讲。比如一个步骤同时用了 Express、CORS、dotenv，
但其中只有 CORS 是新的 → 只讲 CORS。同一功能内多次用到同一技术 → 只讲一次。

---

## 七、边界情况

- **技术名写错了/想改名**：用户说"技术索引里那个 xxx 改成 yyy"→ 照做，同时更新索引对应行
- **用户主动问"这个是什么原理"**：这是用户在追问，不是阈值 1 触发。正常深入解释，不受两句封顶限制
- **不确定功能叫什么**：用最简短的中文描述该功能，够区分就行，不需要精确命名
- **多个新技术同时出现**：每个技术单独处理。同时引入 Express 和 SQLite → 如果都是新的，各输出两句，索引中各加一行
- **用户想重新启用触发引线**：用户说"帮我加上触发引线"→ 重新执行首次启用流程中的步骤 2~4（确定家目录 → 确定全局配置文件 → 告知并写入），但跳过步骤 1 和 5（不重复询问、不重建 `.inverter-setup` 标记）
